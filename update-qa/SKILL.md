# update-qa — 文本整理入库 Q&A

将用户输入的一大段文本（学习笔记、issue 讨论、群聊记录、博客摘录等）整理成标准 Q&A 对，**先经源码验证确认正确，再经用户确认后**追加到 `qa/KNOWLEDGE.md`。与 `qa` 的区别：`qa` 是读 Q&A 库回答问题，`update-qa` 是写 Q&A 库。

> 入口：`/mooncake-agent-skills update-qa "<一大段文本>"`（入库模式）或 `/mooncake-agent-skills update-qa audit [<主题>]`（审计模式）
> **无参数时**：使用 `AskUserQuestion` 展示模式选项——「粘贴文本入库」（默认，选中后提示粘贴文本）| 「audit 审计已有条目」（含 "Other" 供自由输入），不静默退出。

## ⚠️ 核心原则：先验证，后入库

未经源码验证的内容**不允许**写入 KNOWLEDGE.md。Q&A 库是 `qa` 子命令的回答依据，错误条目会被后续回答放大传播。宁可拒绝，不可存疑入库。

## 执行流程

### Step 1: 提取候选 Q&A

1. 从文本中识别问题-回答结构，一段文本可拆出多条 Q&A
2. 每条候选包含：
   - **问题**：改写为简洁疑问句（与已有 `### Q:` 条目风格一致）
   - **答案**：一句话结论 + 必要细节，风格与已有条目一致
   - **目标主题**：匹配 KNOWLEDGE.md 已有的 `##` 二级标题；无匹配则建议新主题（在预览中标注「新建主题」）
3. 从答案中提取**可验证声明**清单（配置默认值、函数行为、错误原因、性能特性、版本差异等），作为 Step 2 的验证点
4. 与 KNOWLEDGE.md 已有条目**查重**：
   - 完全重复 → 丢弃，预览中标注「已存在」
   - 部分重叠但新答案更完整/纠正旧条目 → 标记为「更新」而非新增

### Step 2: 源码验证（审查上下文要求与 review 相同）

对每条候选 Q&A 的每个可验证声明，在 Mooncake 源码（`$MOONCAKE_REPO`，默认 `~/src/Mooncake`）中验证：

1. **定位相关代码**：使用 Grep/Glob/Read 找到声明涉及的函数、配置、常量定义
2. **读取完整函数**：从函数签名读到结尾，不允许仅凭 grep 命中片段下结论
3. **追踪调用链**：至少向上追踪 1 层（谁调用）和向下追踪 1 层（调用谁），确认声明在真实调用路径中成立
4. **交叉文件关联**：声明涉及跨文件交互时（如 client ↔ master ↔ storage backend），读取所有相关文件的交互路径
5. **标注证据层级**：每个声明标注验证深度——
   - `[func-context]` 读完整函数后验证
   - `[call-chain]` 追踪调用链后验证
   - `[doc-only]` 仅有官方文档/注释佐证，源码未直接体现
   - `[unverifiable]` 无法用源码验证（社区经验、外部工具用法等）——不因此直接拒绝，但必须在预览中明示

### Step 3: 对抗反驳（double check，与 review Step 4 相同）

对每条通过 Step 2 的候选 Q&A，启动**独立 agent 尝试反驳**：

1. 答案是否与源码某处逻辑矛盾？
2. 是否只在特定版本/配置下成立，而答案写成了普适结论？
3. 是否存在调用路径使答案描述的行为不发生？
4. double check agent 必须读取比 Step 2 更广的上下文（调用者的调用者、所有子函数、相关头文件/类型定义）

**输出裁决**（置信度 0-100）：

| 裁决 | 含义 | 处理 |
|------|------|------|
| **confirmed** | 反驳失败，答案与源码一致 | 进入预览 |
| **corrected** | 大方向对但细节有误（如默认值、路径不对） | 给出修正后的答案进入预览，标注修正点 |
| **rejected** | 与源码矛盾，或声明的模块/行为不存在 | 不入库，预览中说明反驳理由 |

**仅 confirmed / corrected 且置信度 ≥ 80 的条目允许进入 Step 4**。全部被拒绝时诚实报告——不凑数入库。

### Step 4: 预览与用户确认

**必须**先展示预览表格，再获取用户确认：

```markdown
| # | 问题 | 目标主题 | 裁决 | 置信度 | 证据 |
|---|------|---------|------|--------|------|
| 1 | G2 满了驱逐间隔怎么调？ | 配置参数 | confirmed | 92 | mooncake-store/src/master_service.cpp:L340 [call-chain] |
| 2 | Master 租约默认多久？ | 配置参数 | corrected（原文 30s，实际 10s） | 88 | types.h:L52 [func-context] |
| 3 | ...（rejected 的条目也要列出，附反驳理由） | — | rejected | — | 未在源码中找到该模块 |

拟入库 N 条（新增 X / 更新 Y），拒绝 M 条。
```

使用 `AskUserQuestion` 确认：

```
question: "确认将以上 N 条 Q&A 写入 qa/KNOWLEDGE.md？"
options:
  - "全部入库" (Recommended)
  - "部分入库"（用户说明保留哪几条后再执行）
  - "放弃"
```

### Step 5: 写入 KNOWLEDGE.md

仅在用户确认后执行：

1. 追加到对应 `##` 主题下（新主题则新建二级标题），使用 `### Q: <问题>` + 答案格式，风格与已有条目一致
2. 「更新」类条目：原地修改旧条目，不重复追加
3. 每条 AI 入库条目末尾附来源注记：
   ```markdown
   > 来源：update-qa AI 整理，已通过源码验证（YYYY-MM-DD，证据: <file:line>）
   ```
4. 之后走根 SKILL.md 的后置操作（变更摘要表格 → AskUserQuestion 确认 → commit + push），commit 信息格式：`"knowledge(qa): add <N> Q&A entries via update-qa"`

## 边界情况

| 场景 | 行为 |
|------|------|
| 文本中提取不出任何问题-回答结构 | 告知用户，建议改写文本或直接提供 Q/A 形式 |
| 所有候选均被 rejected | 诚实报告每条的反驳理由，不写入任何内容 |
| 声明涉及源码中不存在的模块/函数 | rejected，说明「未在 $MOONCAKE_REPO 中找到」 |
| 文本与 Mooncake 无关 | 告知本 skill 仅维护 Mooncake Q&A 库，不入库 |
| `$MOONCAKE_REPO` 路径不存在 | 提示用户先 clone Mooncake 源码（见 config.md），验证无法进行时**不入库** |
| 声明全部为 [unverifiable] | 在预览中明示「无源码证据」，由用户显式确认是否仍要入库 |

# audit 模式 — Q&A 库防腐审计

> 入口：`/mooncake-agent-skills update-qa audit [<主题>]`

**动机**：入库时的源码验证是一次性的，Mooncake 源码持续演进，KNOWLEDGE.md 条目会像注释一样腐烂（entry rot）——写入时正确的默认值、函数行为、文件路径可能已经失效。借鉴 Jitskit(arXiv'26) 的 Auditor 设计：定期全量审读实现 vs 规范，把失效点显式化并修复，而不是等 `qa` 回答时把过期信息传播出去。

## 审计流程

### Step A1: 圈定审计范围

1. 无 `<主题>` 参数 → 审计全库；有参数 → 只审计该 `##` 主题下的条目
2. **优先审计带来源注记的条目**（`> 来源：update-qa AI 整理...` 含 file:line，天然是重验入口）
3. 人工维护的条目：凡包含可验证声明（配置默认值、函数名、路径、行为描述）也纳入；纯概念性条目跳过
4. 全库条目较多时（> 30 条），分批执行并告知用户进度，不静默截断——本次未覆盖的条目明确列出

### Step A2: 逐条重验

对每条纳入范围的条目，验证方法与入库 Step 2 相同（完整函数 + 调用链 ±1 层 + 交叉文件）：

1. 来源注记中的 file:line 是否仍存在？函数/配置是否被移动、重命名、删除？
2. 答案中的每个可验证声明在**当前**源码下是否仍成立？
3. 对「已过期」候选执行对抗反驳（同入库 Step 3）：确认是真过期，而不是代码移动了位置但行为未变

### Step A3: 裁决

| 裁决 | 含义 | 处理 |
|------|------|------|
| **仍有效** | 声明在当前源码下成立 | 刷新来源注记中的验证日期 |
| **需修正** | 大方向对但细节失效（默认值变了、文件移动了） | 给出修正稿，预览中标注差异 |
| **已过期** | 行为已不存在或根本改变 | 建议删除或重写，预览中说明证据 |
| **无法验证** | 相关代码找不到、声明不可验证 | 标注，交用户裁定 |

### Step A4: 预览与确认

```markdown
| # | 条目 | 主题 | 裁决 | 变化点 | 证据 |
|---|------|------|------|--------|------|
| 1 | G2 驱逐检查间隔怎么调 | 配置参数 | 需修正 | 默认值 10s → 5s | master_config.h:L88 |
| 2 | ... | ... | 仍有效 | — | ... |

共审计 N 条：仍有效 A / 需修正 B / 已过期 C / 无法验证 D。
```

使用 `AskUserQuestion` 确认：「应用全部修正」(Recommended) / 「部分应用」 / 「仅刷新验证日期不改内容」 / 「放弃」。

### Step A5: 应用变更

仅在用户确认后执行：修正/删除对应条目，刷新来源注记验证日期；commit 信息格式：`"knowledge(qa): audit <N> entries (<A> valid / <B> corrected / <C> stale)"`，走根 SKILL.md 的后置提交流程。

## 审计边界情况

| 场景 | 行为 |
|------|------|
| 全库无带来源注记且无可验证声明的条目 | 告知「无可审计条目」 |
| `$MOONCAKE_REPO` 不存在或版本过旧 | 提示先同步源码，不做任何修改 |
| 用户选「仅刷新验证日期」 | 只更新注记日期，不改答案内容 |

## 维护规则

当本 SKILL.md 有实质性更新，必须同步检查并更新根 `README.md` 中的能力表和项目结构。
